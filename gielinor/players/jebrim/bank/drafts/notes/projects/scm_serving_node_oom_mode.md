# SCM has a THIRD OOM mode: the serving node's uncapped DuckDB (cgroup OOM → 502)

Source: [[S146_f20d7744_scm-serving-memory-review|S146]] (2026-06-02). Extends [[scm_nextjs_duckdb_oom_modes]] (which covered the two *pipeline/DAG* OOM modes). This one is the **serving node** (Next.js web process), and it's the cause of the intermittent **502s**.

`shipping_costs_monitoring_nextjs` is two runtimes: the batch **pipeline** (DAG, separate Airflow pod) and the **serving** Next.js app (EKS deployment `shipping-costs-monitoring`, ns `shipping-dashboard`). The pipeline OOMs ([[S069_006248ef_pipeline-oom-hardening|S069]]/[[S097_1ce9fc1f_scm-daily-product-temp-spill-oom|S097]]) are RAM vs temp-spill. The serving OOM is distinct:

**Serving OOM mode (the 502):** the in-process DuckDB `:memory:` connection (`src/lib/db.ts` `getConn`) set **no `memory_limit`**. DuckDB's native allocator defaults to ~80% of the *host node's* RAM (not the pod's), and lives **outside** the V8 heap — so `node --max-old-space-size=512` (entrypoint.sh) does nothing to contain it. One heavy `read_parquet` over the processed glob (~241MB) overruns the pod cgroup → SIGKILL (exit 137) → in-flight requests **502**. Intermittent because it only fires when a heavy query (or two concurrent) lands.

**Compounding heap drivers (Node side, in the 512MB heap):** the query cache evicted by entry-count not bytes (retained a multi-GB export array 60s); `/api/export` materialized an unbounded per-shipment SELECT + full CSV string + cached it; `convertBigInts` deep-cloned every result; `bd_cache` (breakdown) materialized a wide per-shipment slice into a resident temp table with a 2024–2099 no-date fallback; several routes silently full-scanned the processed glob with no date bound.

**Fix shipped ([[S146_f20d7744_scm-serving-memory-review|S146]], branch `fix/scm-serving-memory`):** cap the serving DuckDB conn (`memory_limit`+`temp_directory`+`threads`+`preserve_insertion_order`, env-tunable) — the crash fix, makes a wide scan spill/cap instead of OOM-killing the pod; byte-aware cache; in-place BigInt convert; export COUNT+413 cap; outliers LIMIT; prune the processed scans (avg-costs/completeness windowed, trends routes via `dailyTierExpr`, breakdown bd_cache window bounded).

**Key lesson (generalizes):** an in-process embedded analytics engine (DuckDB, etc.) in a memory-limited container is a **second, uncapped memory pool** beside the language heap. Capping the language heap (`--max-old-space-size`) is necessary but not sufficient — the engine's own `memory_limit` must be set to the pod budget, or it sizes to the host and OOM-kills the pod. **Deploy-side:** the k8s serving manifest (mem limit, `/data` volume type, probes) is not in the app repo — it's the missing half; `DUCKDB_MEMORY_LIMIT` must be sized to it.
