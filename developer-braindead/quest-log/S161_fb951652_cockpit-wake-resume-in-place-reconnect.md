# [[S161_fb951652_cockpit-wake-resume-in-place-reconnect|S161]] — 2026-06-04 — Cockpit wake-resume: in-place reconnect (fix [[S160_fb951652_cockpit-clickable-paths-and-sleep-wake-resume|S160]] owned-resurrection)

Reopened post-[[S160_fb951652_cockpit-clickable-paths-and-sleep-wake-resume|S160]]-close (same sid `fb951652`); OPEN→CLOSING via `comms_append`; live parallel Jebrim session present (player work, no overlap). Niklavs caught a side effect of [[S160_fb951652_cockpit-clickable-paths-and-sleep-wake-resume|S160]]'s wake-resume: after a wake/relaunch, **six old sessions popped up as "0s / YOUR MOVE / running in this cockpit"** that he hadn't touched.

- **Diagnosis.** [[S160_fb951652_cockpit-clickable-paths-and-sleep-wake-resume|S160]]'s wake handler did `location.reload()`. Reload wipes in-memory state, and the load-time resume-on-open effect ([[main|cockpit/web/main.js]]) then `claude --resume`s the **entire persisted owned set** (`localStorage cockpit-owned-terms`), which is **never pruned on natural session end** ([[term|cockpit/web/term.js]] `ws.onclose` only prints "[session ended]", doesn't `removeOwned`). So every session the cockpit ever launched resurrected — fresh-launched (0s) and idle-awaiting-input (YOUR MOVE). Pre-existing resume-on-open behavior, but [[S160_fb951652_cockpit-clickable-paths-and-sleep-wake-resume|S160]]'s reload made it fire on every wake. I'd flagged the compose-text + survived-connection trade-offs at [[S160_fb951652_cockpit-clickable-paths-and-sleep-wake-resume|S160]] close but **not** this owned-resurrection one.
- **Fix (Niklavs picked "revive only what was open").** Dropped the reload from the wake path. New `TermConn.reconnect()` ([[term|cockpit/web/term.js]]): force-close the dead/half-open socket (so ptybridge's `finally` is sure to terminate the old claude — no double-writer), then after a 1s settle re-open the WS in resume mode (`claude --resume` the same `sessionId`) on the **existing** xterm + board identity. New export `reconnectTermsAfterWake()` iterates only the `live` map (terminals actually open in this page) and reconnects each. [[main|cockpit/web/main.js]] wake-gap `useEffect` now calls `reconnectTermsAfterWake()` instead of `location.reload()`. No reload → the `live` map is preserved → resume-on-open is **not** re-triggered on wake, so old/closed owned sessions stay gone (the full owned-set resume is now a genuine-reopen-only thing). Half-open "can't type" case is still handled (we force-close before resuming, regardless of the socket's reported readyState).
- **Verification.** `node --check` clean on both files. **RUNTIME-UNVERIFIED beyond syntax** — `cockpit/web/term.js` can't be imported in Node (needs `window`/xterm UMD globals), and a physical sleep/wake can't be reproduced in-session. The real test is Niklavs' relaunch → sleep → wake: only the *open* terminals should reconnect in place, no old ones popping back. Same posture as prior cockpit work ([[S142_a31a2bc0_brain_health_audit_and_gitignore_hygiene|S142]] retired cockpit runtime-verification to the eyeball).
- **Learning harvested** ([[build-lessons|bank/build-lessons.md]] +1, + cross-conv memory): a blunt `reload`/reset re-runs **all** restore-from-persistence, not just live state — when you mean "restore what was open," revive the in-memory live objects, don't reload.
