# S023 ‚Äî 2026-05-22 ‚Äî Visualizer ticker fix + cross-session attribution gating

- Diagnosed and fixed the "stuck on 20:37" ticker: `updateTicker()` in live mode was only reachable via `spawnDwarf` / `despawnDwarf` / `spawnGnome` / `despawnGnome` and the `tick()` loop's `if (playing)` branch ‚Äî but `playing` never goes true in LIVE mode (autoplay block at 2897 short-circuits with `if (LIVE) return`). Last refresh was the D7 despawn at UTC 17:37:18 during [[S022_visualizer_audit_fixes]], so the ticker had been frozen for hours while the underlying `currentEventWallTime` ticked forward. First patch added `updateTicker()` after each `pollLive` chunk; that fixed event-driven refresh but the principal pointed out the clock should keep ticking between events. Switched live-mode display to browser wall clock (`new Date().toISOString()` ‚Üí `formatWall`) and added a 1s `setInterval(updateTicker, 1000)` so the time advances every second regardless of stream activity.
- Diagnosed the inverse of [[S022_visualizer_audit_fixes]]'s cross-session Bash attribution: S022 chose recency over `active-mode.txt` because the marker bled into parallel sessions; but with both sessions active and writing intent, the recency walk picked whichever session wrote *last*, regardless of which session's tool call was firing the hook. Mine got tagged as Jebrim. Stamped every event with `sessionId` from the PreToolUse / PostToolUse payload (`_SESSION_ID` module global set in `main()`; `append()` adds the field automatically), made `_walk_recent_actor()` filter by `session_id` when given one, and threaded `_SESSION_ID` through `current_main_actor()` and `infer_dwarf_parent()`. Verified live ‚Äî my next Bash attributed correctly to Braindead.
- Found a second cross-session attribution bug while the principal was watching: Jebrim's wildcard-glob calls (e.g. `gielinor/players/*/quest-log/*`) had no literal `/players/jebrim/` substring, so `classify()` returned `defaultActor=wisp`; the existing dev-brain override then flipped `wisp ‚Üí braindead` because `active-mode.txt == "dev-brain"`. So Jebrim's wildcard scans drove *Braindead's* sprite around the dev-brain session's map while Jebrim sat parked at quest-hall. Introduced `is_dev_brain_session()` (active-mode is dev-brain AND `state-actors.json._mode_session_id == _SESSION_ID`), stamped `_mode_session_id` in `handle_active_mode_write` on both mode-transitions and same-mode reconciliation writes, and swapped all three dev-brain-override callsites (`handle_write_or_read`, `current_main_actor`, `infer_dwarf_parent`) to use the gated check.
- Added a session-filtered recency fallback for wildcard paths in `handle_write_or_read` so a non-dev-brain session's wildcard glob attributes to whichever player that session has been operating as, picked up from its own stamped intent/move events. Without this, Jebrim's wildcard globs would now correctly *not* be braindead but would attribute to wisp instead ‚Äî better than the old bug but worse than necessary. Parallel pattern to `current_main_actor()`'s Bash fallback.
- Branded as a three-incident pattern: shared global state at brain root (`active-mode.txt`, the recency walk over a shared `state.ndjson`) is hostile to parallel Claude sessions. [[S022_visualizer_audit_fixes]] noted this for Bash; [[S023_visualizer_ticker_and_cross_session_attribution]] proves it again for the dev-brain override and for wildcard-path actor inference. The mitigation in both cases was the same ‚Äî gate every shared-state read on `_SESSION_ID`. Worth a `bank/decisions/` note next bankstanding: *parallel-safe attribution requires every shared-state check to be session-gated*.

**Cascade.** `developer-braindead/experiments/visualizer/index.html` (updateTicker call in pollLive, live-mode display switched to wall clock, 1s setInterval); `developer-braindead/.claude/hooks/emit-event.py` (module-global `_SESSION_ID`, `append()` stamping, `is_dev_brain_session()`, session-filtered `_walk_recent_actor`, session-gated overrides in `handle_write_or_read` / `current_main_actor` / `infer_dwarf_parent`, `_mode_session_id` reconciliation in `handle_active_mode_write`).
