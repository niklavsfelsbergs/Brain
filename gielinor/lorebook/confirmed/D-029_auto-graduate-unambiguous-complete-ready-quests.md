# [[D-026_graduate-complete-ready-quests-in-session|D-026]]'s graduation rule is documented but doesn't fire under rapid same-terminal handoff chains

**Observation ([[B-010_2026-05-29_tenth-bankstanding|B-010]], 2026-05-29).** Jebrim's `quest-log/in-progress/` held **22 files** at the start of this bankstanding — the largest backlog yet, the entire [[S114_db60ed8a_eu-tender-austrian-post-rebuild|S114]]–[[S122_330dea7d_eu-tender-switchable-incumbent-open-qs|S122]] EU-tender campaign plus its dwarf/penguin traces, all complete-ready, all graduating in one Phase 0 pass (22→0). This is the **third occurrence** of the deferred-graduation leak and it is **getting worse**, not better: [[B-004_2026-05-23_fourth-bankstanding|B-004]] (15→3), [[B-007_2026-05-25_seventh-bankstanding|B-007]] (15→5), [[B-010_2026-05-29_tenth-bankstanding|B-010]] (22→0).

**Why this is notable despite [[D-026_graduate-complete-ready-quests-in-session|D-026]] + the ritual fix already being in place.** [[D-026_graduate-complete-ready-quests-in-session]] decided "graduate in the session that finishes the quest," and the follow-up ritual edit *was* applied — `spellbook/rituals/close-session.md:125` now states "'Leaving open: propose →completed/ next session' is no longer an acceptable CLOSING line." So both the decision and its ritual enforcement exist. The leak persisted anyway, and worsened.

**Root cause (refines [[D-026_graduate-complete-ready-quests-in-session|D-026]]'s).** The close-session scan *does* fire — the comms CLOSINGs verbatim surface the candidates for approval (e.g. [[S118_f41737e5_eu-tender-decision-scorer-report-regen|S118]]'s CLOSING: *"Proposed complete-ready → completed/ (awaiting y/n): [[S114_db60ed8a_eu-tender-austrian-post-rebuild|S114]] / [[S115_db60ed8a_eu-tender-dpd-pl-reply-review|S115]] / [[S117_d1a3b803_eu-tender-dpd-pl-gls-engine-builds|S117]] / [[S118_f41737e5_eu-tender-decision-scorer-report-regen|S118]]"*). The break is at the **approval gate under rapid iteration**: in fast same-terminal handoff chains ([[S117_d1a3b803_eu-tender-dpd-pl-gls-engine-builds|S117]]→[[S118_f41737e5_eu-tender-decision-scorer-report-regen|S118]]→[[S120_3760e65b_eu-tender-full-year-build|S120]]→S121, each "adopting the prior session's checkpoint, same terminal, no fresh respawn"), the principal is heads-down — the next instruction is "build it" / "push it" / "next step," and the per-quest graduation y/n gets deprioritized and re-deferred to the following session, which does the same. [[D-026_graduate-complete-ready-quests-in-session|D-026]] made *move* the default action but kept it gated on a principal approval that a heads-down iteration loop doesn't supply.

**Why it matters.** Same as [[D-026_graduate-complete-ready-quests-in-session|D-026]]: a deep in-progress folder defeats the crash-recovery signal (`death-and-spawn.md`) and noises up every respawn survey. But the new datum is that the **documented-rule-plus-ritual-step layer was not sufficient** — the leak is structural to the rapid-handoff mode, and bankstanding Phase 0 is doing the entire graduation job as the backstop-become-primary, exactly what [[D-026_graduate-complete-ready-quests-in-session|D-026]] said it shouldn't be.

**Decision (principal-chosen at [[B-010_2026-05-29_tenth-bankstanding|B-010]]): auto-graduate the unambiguous case.** When a CLOSING records deliverable **shipped + committed + no named open dependency**, the *move* executes in that close **without a separate y/n** — reported as a vetoable batch; the principal reverses after the fact if wrong (a `git mv` back, zero loss under the no-deletes guarantee). Anything with a stated open dependency or any ambiguity stays propose-and-confirm. This removes the approval gate at `close-session.md:123` ("propose only… principal approval, every time") that the rapid-iteration loop skips — the actual leak, since [[D-026_graduate-complete-ready-quests-in-session|D-026]]'s default-direction wording was already in place and didn't hold.

The ritual implementation is drafted as a godly proposal: `deities/guthix/proposals/2026-05-29-auto-graduate-unambiguous-complete-ready-quests.md` (concrete `close-session.md` step-4 edit).

**Alternatives rejected:** (b) handoff-adopt graduates the predecessor — narrower, misses cross-terminal debris; (c) leave Phase 0 as the de-facto mechanism — concedes the crash-recovery cost [[D-026_graduate-complete-ready-quests-in-session|D-026]] named; (d) more wording on the default-is-move rule — [[D-028_grounding-precondition-needs-a-trigger|D-028]]'s lesson is that a rule which isn't firing needs a trigger, not another note.
