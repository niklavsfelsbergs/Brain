From [[S038_vscode_claude_focus_extension]] (new): **`claude_pid_chain` is the right matching surface, not a single PID.** VS Code's `terminal.processId` exposes the shell PID (chain depth ~3 on this machine). Hook fires from deeper. Matching against the whole chain absorbs the depth asymmetry. The chain-capture-at-fire-time pattern from [[S037_terminal_switchboard_phase_3_click_to_focus|S037]] paid off again, for a different reason — generalize: *if you'll need to match a process tree later from a possibly-distant vantage point, capture the whole chain at write time.*

From [[S037_terminal_switchboard_phase_3_click_to_focus]] (new): **Windows PowerShell 5.1 reads `.ps1` files as cp1252 unless a UTF-8 BOM is present.** A single em-dash in a `Write-Log` string broke the parse — the three UTF-8 bytes got read as three cp1252 characters, swallowing the closing quote and cascading errors through the rest of the file. **Discipline: pure ASCII for `.ps1` scripts, or write with UTF-8 BOM.** Worth a lorebook draft paired with the hook-fire-rate-as-budget lesson (both Windows substrate gotchas).

From [[S037_terminal_switchboard_phase_3_click_to_focus]] (new): **walking the process tree at hook fire time is the right shape for "I need an ancestor PID later" on Windows.** `os.getppid()` from a hook subprocess points at a short-lived wrapper (two layers of `bash.exe` on this machine) that exits seconds after the hook returns. `Get-Process -Id <dead>` returns nothing; PIDs may get recycled. Capturing the full ancestor chain *while everyone is still alive* via `CreateToolhelp32Snapshot` sidesteps all of it. Same pattern fits other instrumentation that needs to retain process-tree facts across time.

## Build-hygiene invariants (Khaan benchmark, item 12 — [[S115_a9fe4749_khaan-failure-receipts-and-hygiene|S115]], 2026-05-28)

Three codified engineering rules, lifted from the Kha'an brain (plan §P / item 12) and confirmed to match what we already do — written down so the autonomy phase inherits them as stated invariants, not folklore:

- **Atomic-write all state JSON.** Anything a hook/cockpit/sidecar reads while another process may be writing it (`switchboard/state-*`, `~/.claude/status/*.json`, intent sidecars, names stores) is written `tempfile + os.replace`, never an in-place truncate-then-write — a reader must never catch a half-written file. **The one trap: never atomic-write onto a *symlinked* path** — `os.replace` replaces the link with a real file and severs it. Live risk only at the ascension/VPS phase, if the brain is ever symlinked onto a host.
- **Ship dormant.** A new capability (skill, script, scheduled job) **never self-wires** its own cron / auto-invocation / always-on trigger. The principal promotes it from dormant to live. This is literally our HITL ethos stated as an engineering rule — and the strongest external validation (an autonomous brain independently arrived at the same brake) that the instinct is right. *(Pairs with the "don't over-formalize a reachable capability" memory: ship the dormant minimum, let real use pull it live.)*
- **Minimal surface area.** No speculative `--dry-run` / `--force` / config flags added "in case." Add a flag the first time a real call needs it, not before. Every unused option is a maintenance + test-surface tax. *(`born-link-lint.py`'s `--check` exists because the pre-commit shim actually calls it; that's the bar.)*
