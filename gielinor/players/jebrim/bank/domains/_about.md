# bank/domains/ — Jebrim's work-domain digests

> **What this layer is.** A player's *standing, synthesized knowledge of their own work domains* — so the player **just knows their topics** instead of retrieving the right note on the right keyword. Born §Z (designed [[S177_988f0608_brain_health_check_procedure_and_first_rating|S177]], built S179+; see `developer-braindead/bank/plan.md` §Z). This is an **identity** statement, not just a path: a player's `bank/domains/` *is* "the work I know." SCM is Jebrim's; Latvia-property is Zezima's.

## The problem it solves

`bank/notes/` holds the *facts* (draft → bank → keepsake is the fact ladder), but **topic-routing had no ladder** — a domain was all-or-nothing manual (a `cue_registry.py` row), which is why SCM / EU-tender / carrier-contract sat *named-but-unbuilt* for weeks. A prompt like *"let's work on the SCM"* fired **no** retrieval: `SCM` matched no cue pattern, the keepsake force-read carries mart-not-SCM pins, and the ~6 real SCM notes loaded only on the model's discretion. The brain *had* the knowledge; nothing forced it on-topic.

The fix is the brain's own `research → bank` *picking* pattern, applied one level up. At scale you can neither inline nor name 100 notes — so the raw note **corpus** stays the anchor (read on demand), and a synthesized **digest** is what gets loaded. A domain digest is, in effect, a **per-domain keepsake**: distilled, budgeted, regenerated by alching.

## The two tiers (per domain)

- **Tier 0 — the digest** (`<topic>.md`, ~2 KB): the load-bearing claims + a map of the corpus + source links. **Cue-loaded** — force-inlined when its topic fires (the §Z.C auto-discovery arm), zero eager cost otherwise.
- **Tier 0 — the index** (`_index.md`): the *map*, one line per domain. **Always-read** every session (force-injected like the keepsake), so the player always knows the lay of their land. Small and budgeted.
- **Tier 1 — the corpus** (`bank/notes/projects/*.md`): named in each digest, read on demand. The digest says *which* note carries the detail.
- **Tier 2 — the specialist / live source**: spawn (the shipping-agent, the live mart). The digest points; it never restates the live source.

**Never all-digests-always-on** — that rebuilds the always-on bloat the §X `@import` trim fought. Index always-on (a map); digests on topic.

## Digest frontmatter schema

Every digest carries this frontmatter. The §Z.C loader **auto-discovers** the active player's `bank/domains/*.md`, reads `patterns`, and inlines the matching digest — so **adding a domain = alching drops a digest file** (no hook/registry edit).

```yaml
---
domain: scm                         # short slug, unique within the player
title: Shipping Costs Monitoring …  # human one-liner (shown in _index.md)
patterns:                           # cue strings; a prompt matching any → inline this digest
  - scm
  - shipping costs monitoring
  - shipping_costs_monitoring_nextjs
corpus:                             # the Tier-1 anchor notes, paths relative to the player root
  - bank/notes/projects/shipping_costs_monitoring_nextjs_vocab.md
  - …
specialist: null                    # optional: a sub-agent / external repo to spawn for live work
freshness: 2026-06-09               # the date the digest content reflects (≈ max corpus date)
synthesized: 2026-06-09             # the date alching last (re-)generated this digest
---
```

- **`patterns`** are the cue strings the §Z.C arm matches against the prompt (case-insensitive substring). Keep them tight — over-broad patterns inline the digest on unrelated turns.
- **`corpus`** paths are relative to the player root (`players/<player>/`). Alching compares each file's mtime against `synthesized`; a corpus file newer than `synthesized` is the **re-synthesis signal**.
- **`freshness`** lets a reader (and the `freshness` drift-check) judge staleness without opening every source.
- **`specialist`** routes live work — e.g. SCM points at the shipping-agent for the live mart, since the digest is *about* the dashboard, not a substitute for querying it.

## Loader convention

Files whose name **starts with `_`** (`_about.md`, `_index.md`) are infrastructure, **not digests** — the auto-discovery glob skips them. A real digest is `<domain-slug>.md` with the frontmatter above.

## Budget

- **Digest:** ~2 KB (force-inline byte cap `INLINE_BYTE_CAP` still applies). A domain whose load-bearing surface won't fit in ~2 KB is the signal to **split the domain**, not to grow the digest.
- **Index:** one line per domain. The index is a map, never a digest-of-digests.

## Lifecycle (alching owns it)

- **Born.** The §Z.D domain-coverage detector flags a note-cluster (or keepsake work-pin) with no digest → alching synthesizes one.
- **Tended.** Alching re-synthesizes a digest when any `corpus` file's mtime is newer than `synthesized`.
- **Surfaced.** The index is always-on; the digest fires on its own `patterns`.

Synthesis is a **drafted, approved output** but a **light gate** — advisory context, fully source-linked (a gnome can draft, the principal spot-checks; it is *not* a `confirmed/` identity write). The source links make every claim verifiable and catch drift.

## Relationship to the global cue_registry

The global `gielinor/.claude/hooks/cue_registry.py` is **retained for external / specialist domains** (shipping → the external `picanova/shipping-agent` repo + the shipping-agent sub-agent). Per-player `bank/domains/` is the new primary for a player's **own** topics, auto-discovered from digest frontmatter. A domain can touch both — Jebrim's `scm.md` digest *and* a `specialist:` pointer to spawn the shipping-agent for the live mart.
