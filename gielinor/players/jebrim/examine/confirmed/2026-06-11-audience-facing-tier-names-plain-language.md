# Audience-facing category names must be plain-language, not analyst vocabulary

**Anchor ([[S194_907d4e63_eu-tender-holdup-pass-and-project-rating|S194]] / sid8 907d4e63, 2026-06-11).** Proposing the final-report saving tiers I named one "firm-unvalidated" — internally precise (engine has no own-actuals overlap). Niklavs: *"unvalidated sounds like its wrong. we need to make the names more understandable for someone not working on the project."* Renamed to Confirmed / Offer-based / Conditional, each label answering "where does this number come from" instead of grading its epistemics.

**The rule.** Names in a deliverable for readers outside the project carry the reader's connotations, not the project's definitions. Analyst-jargon labels (unvalidated, low-confidence, face-value, contingent-on-X) read as defects or hedges to an outsider. Name the *source or condition* ("offer-based", "pending the package check") rather than the *epistemic grade* — the grade goes in the description column.

**Generalizes.** Sibling of rule-text-becomes-agent-vocabulary (S-2026-05-22): that one is internal-language leaking out; this is internal *taxonomy* leaking out. Check every label in a management-facing artifact against "what would someone who never saw this project think this word means?"
