# Band / filter at entity grain, not on a line that lives on one row of the entity

**Observed ([[S279_6fbdcee1_orwo-tender-roadmap-and-ups-base-validation|S279]], 6fbdcee1).** I reported a "big cheap sub-gram ~€0.97 mail stream" in ORWO UPS invoices and advised segmenting it out of the tender. Niklavs asked "what's the share?" — checking it collapsed the claim: it was **0.4%** (49 trackings), not a stream. Root cause: my example query filtered `WHERE actualweight <= 1` at the **invoice-line** level. A parcel is many charge lines; the *weight* sits only on the freight line, while the fuel/tax/surcharge lines carry weight 0. So the filter **dropped each parcel's freight line** and kept the zero-weight companions — making normal ~6.5 kg parcels look like €0.97 sub-gram mail.

**Lesson.** When an entity (a shipment/parcel/order) spans multiple rows and a property (weight, the base charge) lives on only *one* of those rows, **never filter or band on that property at the row level** — it silently drops the rows that don't carry it and re-shapes the population. Roll up to entity grain first (`GROUP BY tracking`, take `MAX(weight)` / `SUM(freight)`), *then* band/filter. The clean tracking-grain view gave the true picture immediately.

**How to apply.** On any line-item / charge-line / invoice data: aggregate to the entity before any weight/cost cut. If a per-band count looks surprising (a huge ultra-light band), suspect a line-level filter artifact before believing it. Sibling of [[disagreeing-totals-grain-mismatch]] and the count-unit reflexes — same family: name the grain before trusting the number.
