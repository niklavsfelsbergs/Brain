# Validate a refinement against the downstream constraint before recommending it

**Observation (S150, 5aeaa18a, 2026-06-05).** Building the per-carrier overview, I chased a refinement — switching the weight axis from gross to **chargeable weight at fine (1kg) bands** — because it improved the *local* metric (raw fidelity to the packagetype routing, 93%→96%). I recommended it and built it. But the principal's actual requirement was **contiguous (sequential) weight bands per carrier**, and fine bands wobble far more, so enforcing that contiguity *crashed* the result to 84% fidelity and a €123k penalty — strictly worse than the coarse gross-band version (~92.7% smoothed). The 96% was real but it lived *upstream* of the smoothing the principal wanted, and didn't survive it.

**Why it matters.** Optimizing one stage in isolation can be undone — or reversed — by a constraint further down the pipeline. The 96% looked like a clean win on the metric I was watching; I didn't run it through the contiguity smoothing (the genuine requirement) before recommending it. A local-metric gain is a hypothesis until validated end-to-end against what the deliverable actually has to satisfy.

**How to apply.** Before recommending or building a refinement that improves one number, ask "does it survive the downstream step the user actually needs?" and test it through that step first. Here: I should have smoothed the fine-band version *before* presenting the 96%, which would have shown the 84% immediately and saved a detour. Sibling of [[2026-06-01-verify-the-thing-dont-trust-the-wiring]] — a metric improvement is "the wiring"; surviving the full pipeline is "the thing."
