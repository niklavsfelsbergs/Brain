# Live evidence beats a confident chat-side diagnosis — check ground truth when the system is reachable

**Observation ([[S146_f20d7744_scm-serving-memory-review|S146]], 2026-06-02).** Asked "any idea what's going on?" about the Breakdown tab failing, I gave a **confident, code-grounded diagnosis: the `bd_cache` shared-connection temp-table race** — even cited the route's own retry blocks as proof. It was wrong. When I then got cluster access and ran `kubectl describe pod`, the truth was plain: **OOMKilled (exit 137), 3 restarts, 1.5Gi pod limit**, and `grep bd_cache` in the logs returned **nothing** — the race wasn't firing at all. The real cause was the OOM (DuckDB on the 4GB default vs a 1.5GB pod — the exact caveat I'd flagged at ship time, then forgot to lead with).

**Rule.** A diagnosis assembled from reading code is a *hypothesis*, however well-grounded — not a finding. When the running system is reachable (or reachable with creds I can ask for), get the ground-truth signal **before** asserting a root cause: pod state / exit codes / logs / metrics. My bd_cache story was plausible and specific, which made it *more* dangerous, not less. The tell I missed: I had already named the OOM-vs-pod-limit risk in this very quest — the live check should have started there, not at the cleverest code-level hypothesis. Reinforces [[feedback_instrument_dont_reguess]] (instrument, don't re-guess) and the "verify the measurement measures the thing" family — applied here to root-causing, not just fix-params.

Caught + corrected same session (kubectl confirmed OOM → applied `DUCKDB_MEMORY_LIMIT=512MB` → pod stable). No wrong fix shipped, but a wrong *diagnosis* was stated confidently first.
