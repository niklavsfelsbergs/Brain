# S063 — 2026-05-23 — Switchboard feed liveness + the stale-JS cache fix

Principal opened in dev-brain (*"time to work on the switchbar app"*) then **"I need to test it from a clean slate."* Did a clean-slate reset, iterated on feed liveness, and chased a "feed looks dead" report down to its real root (stale browser JS). Session ended with the principal deciding to **rebuild the whole switchboard with a new agent** — so the durable value here is the *learnings*, not the code.

- **Clean-slate reset (reversible).** Archived the bloated event streams (`switchboard/state.ndjson` 1995 lines + `chat.ndjson` 1550 lines, recreated empty), **75 dead per-session status files**, and a stale intent → all to `~/.claude/status/archive/`. Relaunched the server **detached via [[start-switchboard.vbs]]** — confirmed it runs `.venv\Scripts\pythonw.exe server.py` parented to a detached `pythonw` (not any terminal). **This validated the [[S062_switchboard_lifecycle_feed_persistent_server_peek|S062]] VBS launcher, previously never eyeballed.** Board came back to just the 3 live rows (me + 2 live Jebrim sessions) with an empty feed.
- **Feed liveness — the design tension.** Principal: the feed *"doesn't look like it's feeding."* Root: [[S062_switchboard_lifecycle_feed_persistent_server_peek]] silenced raw tool-actions in the feed (clean lifecycle ticker), which made an actively-working session **invisible** — liveness moved into the row sparkline/action-line, the feed itself went quiet. Fix direction (landed, uncommitted): **throttled ambient action stream** in `chat.js` — actions render again but capped at ~1 line/session/2.5s (`shouldRenderAction`), reusing the intact `.act-glyph`/`.act-body` styling. Sim-verified: 10-action burst → 2 feed lines; sessions throttle independently.
- **`done`-per-turn → done-on-close.** A `✓ DONE` fired on every turn-end echoing the last intent — read as "the session keeps finishing." Changed `status-sidecar.py` so `done` fires only at **real close** (the `wrapped_up` Stop, or `SessionEnd` if it ended without wrapping — deduped via the surviving `mode` value so never twice). Per-turn Stop now emits nothing; liveness comes from the action stream + intent beats.
- **THE root cause of the whole "feed is dead" report: stale browser JS.** The board's ES-module imports aren't cache-busted, so a hard-refresh doesn't reliably replace them, and a *live-polling feed updates with whatever JS was loaded at the last real page load*. Proved the data was correct (14 braindead + 3 Jebrim `action` records in `chat.ndjson` in the last 5 min) while the screenshot showed zero — i.e. old chat.js silencing them. **Fixed at the root**: `server.py` static handler now sends `Cache-Control: no-store`. No service worker, plain `app.js` module entry → a *plain reload* always runs latest. **This kills the "hard-refresh or it's stale" tax that has bitten nearly every switchboard session.** Verified the header lands and the new chat.js is served.
- **Graceful intent truncation.** Principal: *"your message is cut off."* Intent capped at `INTENT_MAX_LEN=280` with a hard mid-word slice. `status-sidecar.py:244` now clips at a word boundary + `…`. (Discipline note: I also kept writing intents *over* 280 — the cap was doing its job; the chop just looked broken.)

**Diagnostic lesson (the expensive one).** I misread "Jebrim isn't in the feed" three times by snapshotting `chat.ndjson` over **60–90s windows** while Jebrim was blocked on his dwarves (parent emits nothing while sub-agents run). A 5-minute tally showed his actions *were* flowing. **Don't diagnose liveness from a narrow time window when sub-agents are involved.** Bigger lesson: **a stale-JS environment makes ALL live verification unreliable — fix the cache story FIRST, before iterating on any browser feature.** Four round-trips with the principal were spent on a problem that was 90% stale JS.

**For the rebuild (carry-forward).**
- **Keep the `no-store` fix** (or bake cache-busting into the build) — it's rebuild-agnostic and solves the recurring tax. The new agent should not re-discover this the hard way.
- The **feed-liveness design question is open**: throttled action stream vs. clean checkpoint ticker vs. row-only liveness. Principal wanted the feed to *breathe*; [[S062_switchboard_lifecycle_feed_persistent_server_peek|S062]]'s pure-checkpoint feed felt dead.
- `start-switchboard.vbs` (detached venv server) works — keep the persistent-server model.
- The clean-slate reset recipe is in this entry if the rebuild needs a known-empty board.

**Cascade.** [[status-sidecar.py]] (done-on-close + graceful intent clip), `switchboard/chat.js` (throttled action stream), `switchboard/server.py` (no-store), `comms/active.md` (OPEN + CLOSING), this quest-log entry, `respawn.md`. Clean-slate moves under `~/.claude/status/archive/` (runtime state, gitignored).

**Main-brain changes.** none.
