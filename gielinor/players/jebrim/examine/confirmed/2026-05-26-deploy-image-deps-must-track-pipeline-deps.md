# Deploy-image deps must track pipeline code deps

**Observation ([[S073_006248ef_aws-report-swap-guide|S073]], 2026-05-26).** I deployed the shipping_costs_monitoring cutover + 3 follow-on changes, verified each via `tsc --noEmit` + watching the ECR `:latest` image land, and called them green. The first live DAG refresh then crashed: `ModuleNotFoundError: No module named 'duckdb'` at `pipeline.py` L496 — the [[S069_006248ef_pipeline-oom-hardening|S069]] OOM-fix rewrote the full-pull join to use DuckDB, but the `Dockerfile` only pip-installed `polars pandas pyarrow redshift_connector python-dotenv`. Local `--refresh-full`, `tsc`, and the CICD build all passed because my dev env already had `duckdb`; only the container, on the full-pull path, exposed the gap.

**Pattern.** I treated "tsc clean + image built + `:latest` updated" as "deploy verified." It wasn't — none of those exercise a *runtime* import on the actual deployed path. The dev environment is a superset of the image, so local success is not evidence the image is complete. The incremental path (polars-only) would never have hit it either; it took the full-pull path to surface.

**Rule.** When a code change adds an import, update the deployment manifest (`Dockerfile`/`requirements`) in the *same* change. Before calling a pipeline deploy verified, either run the deployed artifact on the real path or audit its imports against the image's installed deps. Build-success ≠ run-success for runtime imports. Sibling to [[2026-05-22-check-artifact-mtimes-doc-not-source-of-truth]] and [[2026-05-23-inherited-confidence-not-own-confidence]] — verify the actual thing on the actual surface, not a proxy.
