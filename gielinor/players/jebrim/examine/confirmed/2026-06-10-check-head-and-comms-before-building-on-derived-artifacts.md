# In a parallel-session domain, verify current HEAD + comms tail before building on derived artifacts

> Draft self-observation, [[S192_384c1c27_db-schenker-reroute-package-dims]] (sid8 384c1c27), 2026-06-10. Anchor: I ran a whole DB Schenker reroute analysis — package breakdown, savings split, confidence tiers — off `validation/db_schenker/*.parquet` dated 13:00 today, and presented it to Niklavs as "our final routing now." The close-ritual comms read caught that a sibling session had answered the Maersk girth question (L+2W+2H) and rebuilt the engine to 3.2.0 at ~17:25 (commit a96e449), collapsing the Maersk oversize lane: moved population 6,606 → 4,490, switch saving €161.7k → ~€99.7k, Maersk 2,924 → 27. My numbers were stale for the whole session.

**What happened.** I grounded in the topic-matched resume file (`maersk-hermes-oversize-corrections-resume`), which read "awaiting Maersk girth-definition reply." That resume was itself stale — written at the [[S189_e3de8457_maersk-hermes-oversize-corrections|S189]] *first* close (14:45), before the [[S189_e3de8457_maersk-hermes-oversize-corrections|S189]] *continuation* (17:25) received the reply and superseded it. I trusted a derived parquet because it was dated *today* and looked current, and I verified `validation_stats.json` internals — but never checked the engine version or git HEAD until the close ritual forced a comms read.

**Why it's wrong.** In a shared-tree, many-parallel-session domain (EU tender, ~10 sessions/day), *freshness is not implied by a recent file timestamp or a matching resume file*. A later sibling can supersede both. The authoritative current state is **git HEAD + the latest comms CLOSING**, not the resume that happens to match my topic. This is the live-vs-fixture trap ([[distinguish-fixture-from-live]]) and the verify-the-thing reflex ([[verify-the-thing-dont-trust-the-wiring]]) applied to *my own project's derived outputs*: a derived artifact is a fixture until I've confirmed it was regenerated from current HEAD.

**How to apply.** Before building analysis on derived artifacts in a parallel-session domain: (1) read the comms tail's latest CLOSING for that workstream, (2) check the relevant engine/config version or git HEAD, (3) confirm the artifact's provenance matches HEAD — *then* analyze. A resume file is a starting hypothesis about state, not a freshness guarantee. Especially when the principal asks "with our X *now*" — the word "now" is the cue to verify currency, not assume it.

**Credit where due:** the close ritual's forced comms read is what caught this. Argues for pulling that check *earlier* (respawn / first grounding), not relying on close to surface it.
