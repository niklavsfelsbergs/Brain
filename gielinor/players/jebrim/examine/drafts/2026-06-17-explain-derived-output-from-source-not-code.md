# Explain a derived output from the source data, not from the code's apparent intent

**Observation ([[S254_9ced1d5b_eu-tender-ops-report-service-cutoff-rows|S254]], 2026-06-17, sid8 9ced1d5b).** Niklavs asked why the ops routing report showed two DHL services for one DE/WICKELVERPACKUNG cell. I answered confidently from the renderer's logic — the `svc_cell` docstring distinguishes a "size-bound → volume shares" branch, the cell rendered as volume shares, so I explained it as a size-driven product split. **Wrong.** When Niklavs said "check that's actually the case," the parquet showed the packagetype has *constant* dimensions — the split was a clean ~1 kg **weight** cutoff, mislabeled only because routing bands are built on `floor(kg)` and a boundary collision defeated the disjointness test.

**Why it matters.** I explained an *observed output* by reading the code that produced it and trusting the branch it appeared to take — without checking the underlying data the code ran on. The code's branch labels ("size-bound") described an *intent*, not what actually happened for this cell. The cheap verification (group the parcels by service, look at weight vs dimensions) would have caught it in one query before I gave the wrong mechanism.

**How to apply.** When asked *why* a derived/rendered artifact shows something, treat the code's apparent logic as a hypothesis and confirm against the source data before stating the mechanism — especially when a one-line query can settle it. Sibling of [[verify-the-thing-dont-trust-the-wiring]] (global keepsake) and the *check the record before defending a design story* reflex: the failure mode here is the same shape — asserting behavior from how the system *looks* designed rather than from ground truth.
