# Re-scan for late-arriving companion files before concluding a multi-file drop

**Observation ([[S148_104c786b_eu-tender-dhl-paket-round2|S148]], 2026-06-03).** Reviewing the FedEx Round-2 reply, I read the `round_2/` folder when it held only the xlsx + two fuel screenshots, and told Niklavs the #2 lever (RE volumetric divisor) was "still the blocker — not in what landed." It *was* answered — in the covering email `fedex_2`, which had been saved at 11:19, a few minutes after my folder read. Niklavs surfaced it with "did you see fedex.md?"; once I read it, #2 ("all services have 5000 divisor"), fuel scope, and FX were all resolved. My "still the blocker" was wrong because I concluded on an incomplete file set.

**Why it matters.** A carrier reply is not one artifact — it lands as a covering email *plus* attachments (screenshots, spreadsheets), and the pieces can arrive minutes apart. The attachments answer the "show me the data" questions; the email answers the "confirm the mechanic" questions — which is exactly where the load-bearing blockers (vol-weight, scope) get closed. Reading only the attachments and concluding a blocker stands is a one-sided read.

**How to apply.** Before declaring a blocker unresolved on a multi-file drop: (1) re-list the folder right before concluding (files may have arrived since the first `ls`); (2) prefer the covering text/email as the primary source for mechanic-confirmations, not just the attachments; (3) treat a principal nudge like "did you see X?" as near-certain evidence I missed a file — re-scan, don't re-argue.
