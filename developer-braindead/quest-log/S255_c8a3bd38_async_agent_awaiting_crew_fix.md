# S255 — 2026-06-17 — Async Agent spawns never showed "awaiting crew"

- **Reported symptom:** a live session spawned a gnome but the cockpit board never showed "awaiting crew." Diagnosed read-only against ground truth: the gnome (G12, sid 5974b4ee) and a control dwarf (D264, my own session) both **despawned ~230ms after spawn** while the sub-agent ran on for minutes. Transcript confirmed the spawn tool was **`Agent`** (not the old `Task`), with **two `<task-notification>` re-invokes** and **no `run_in_background` flag**.
- **Root cause:** the modern `Agent` tool launches sub-agents **async** — `PostToolUse` fires at *launch* (~230ms, parent gets a handle), not at completion; the real result arrives later via a `<task-notification>` carrying the original `<tool-use-id>`. [[emit-event|emit-event.py]] only recognized async via `run_in_background` (which `Agent` doesn't set), so it treated the spawn as a blocking `Task` and despawned the role-state entry at launch — before the cockpit's ~2s poll could derive the `crew` tag. Same class as the [[S028_subtask_channel_and_guthix|S028]] background-task invisible-sprite bug.
- **Fix (principal chose "persist + clear on notification"):** [[emit-event|emit-event.py]] `handle_task_pre` flags a foreground `Agent` spawn `async_agent`; `handle_task_post` skips the launch-despawn for it (alongside the existing `background` skip) → the entry persists → [[status-sidecar|status-sidecar.py]]'s manifest derives `crew` + "awaiting crew — …". New `_reconcile_async_crew` in [[status-sidecar|status-sidecar.py]] despawns the `async_agent` entry once its `<task-notification>` lands in the transcript (matched by `<tool-use-id>`), emitting the sprite despawn. Routes around the undocumented "does `UserPromptSubmit` fire on a task-notification" question (claude-code-guide couldn't confirm) by reading the transcript status-sidecar already walks; `Stop` guarantees the reconcile runs at turn-end; emit-event's 1h GC is the backstop. `SubagentStop` noted as a future cleaner signal (needs a settings registration + restart, so deferred).
- **Verification:** NEW [[test_async_crew_lifecycle|test_async_crew_lifecycle.py]] **14/14** (persist-through-Post, crew-list inclusion, completion despawn + event, old blocking `Task` still despawns on Post, `run_in_background` Agent excluded from crew, unrelated notification leaves it alone); `test_actor_resolution` 5/5; `cockpit/test_backend` 79/79; `py_compile` clean. **Live end-to-end:** spawned a real `Agent` (D265) → entry **persisted** with `async_agent=True` (vs D264 pre-fix vanishing at +228ms) → on completion the real-transcript reconcile removed D265 and emitted `despawn-dwarf`.
- **Known minor edge** (noted, not over-built): the completion parse keys off transcript lines containing `task-notification` + a `<tool-use-id>`; a pending entry could in theory clear early only if its opaque `toolu_` id were quoted next to "task-notification" before its real notification — impossible naturally (the id isn't known until after spawn). GC + re-spawn semantics cover oddities.
