# A "sibling-class bug" is a per-instance hypothesis — verify each sibling's own spec before asserting it shares the class

**Observed:** [[S190_9a064d86_eu-tender-red-team-audit-and-report-rebuild|S190]] (jebrim-9a064d86), the EU-tender Hermes-rebuild thread.

**What happened.** The oversize fix moved Maersk and Hermes from a per-dimension trigger to a girth-based one (their contracts were genuinely girth-based and the per-dimension read over-fired). I then flagged DHL bulky as the *same* class — "the third sibling of the bug you just fixed twice" — and recommended considering the girth fix for it. Niklavs pushed back: *"we already received a clear answer in round 2."* DHL's round-2 reply explicitly confirms **any-side>60 = Sperrgut even on thin flats** (the 95×65×5 example → "yes, bulky") — so DHL's per-dimension trigger is **carrier-confirmed correct**, not the same defect. Separately, in the same beat I quoted the DHL threshold as `d_max>200` (paraphrased from a docstring) when the code uses `STD_MAX_LENGTH_CM=120` — 200 was the reject cap, not the surcharge trigger; Niklavs caught that too.

**The lesson.** Two layers:
1. **The "fix-the-class-across-siblings" heuristic has an inverse failure mode: over-reach.** A defect verified for two entities is not automatically present in a third — in a multi-entity domain each instance has its own contract. Before raising "this sibling needs the same fix," verify *that sibling's* spec actually shares the class. Here the three carriers split: maersk/hermes girth (fix correct), DHL per-dimension (carrier-confirmed, no fix). This is the same principle as [[feedback_dont_generalize_from_single_verified_case]] / the S150 quadruple-catch — per-instance verification in a multi-carrier domain — applied to a *fix/defect* rather than a *fact*.
2. **Read a constant's value from source; don't paraphrase it from nearby prose.** I lifted "200" from a docstring instead of reading `STD_MAX_LENGTH_CM`. The docstring mentioned both the trigger envelope (120) and the reject cap (200); paraphrasing grabbed the wrong one. Quote the value, not the sentence about it.

**How to apply.** When a fix/diagnosis is about to generalize across sibling consumers of a shared pattern, treat each sibling as a separate hypothesis and check its own contract/spec (or the carrier reply) before asserting it shares the defect — especially before putting it in front of the principal as an action item. And when citing a threshold, read the constant, not the comment.
