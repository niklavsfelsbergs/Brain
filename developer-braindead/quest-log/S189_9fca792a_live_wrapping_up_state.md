# S189 — 2026-06-11 — Live WRAPPING UP state during session-close

- **Follow-on to [[S188_9fca792a_switchboard_state_derivation_fixes|S188]] (same sid8):** Niklavs asked why the board only ever shows WRAPPED UP, never **WRAPPING UP** during a close. Root: the `closing` state is real (sidecar writes the `closing` `.mode` marker at close step 0, backend promotes it to a WRAPPING UP main chip) but never *published* — a clean close runs as **one turn** (write `closing` step 0 → ordinary tools steps 1–10, which don't fire the status sidecar → overwrite with `wrapped_up` step 11 → Stop), so no hook fires *while* the marker reads `closing`, and the board jumps straight to WRAPPED UP. WRAPPING UP only ever showed if the close paused for a nod (a mid-close Stop) — which a pre-authorized close skips.
- **Fix — backend reads the `.mode` marker LIVE per poll** ([[backend|cockpit/backend.py]] `_live_mode_markers` + merge in `build_session_model`), the same pattern it already uses to re-read the action heartbeat fresh instead of trusting the frozen manifest. `closing` → WRAPPING UP within one 2s poll of step 0; `wrapped_up` → WRAPPED UP; ritual flavors publish faster too. No frontend change (the `WRAPPING UP` label already existed in `board.js`).
- **Staleness guard (the catch).** Live-reading exposed **13 lingering `.mode` markers** from past sessions (left on crash-exit, no SessionEnd to archive them) — a naive read flipped *live* rows to a false DONE (e.g. `e9dbce2d`). So a marker is trusted only when its mtime is **newer than the session's `last_event_ts`**: a close's marker is written just after the turn's prompt (valid); a stale marker from a since-resumed session is older than the resume event (ignored). The paused-close case still publishes via the manifest tag the sidecar writes at the question fire.
- **Live-verified through a real close:** session `a17168ec` went `closing` → board WRAPPING UP → `wrapped_up` → WRAPPED UP across its own close while this was being built — the exact lifecycle, observed end-to-end.
