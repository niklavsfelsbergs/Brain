# A class derived from a categorical label needs a physical-reality cross-check

**Observed:** [[S182_e3648d0d_routing-report-size-tiers|S182]] (e3648d0d), routing-report size tiers. I defined the "oversize" size class purely off the `packagetype` string (the families Niklavs named — GEL / CUSTOM_OVERSIZED / zugeschnittene / pallet). It validated cleanly in aggregate. Then Niklavs spotted 24 "oversize" parcels routed to DHL Paket DE and asked "what even are these — looks like a data quality issue."

**The catch:** they were *physically tiny* — ≤22.5 cm longest side, <1 kg — carrying an oversize-sounding packagetype label (a `CUSTOM_OVERSIZED` small custom box, a `GEL` cold-chain envelope). Not measurement DQ (only 5 of 237 had missing dims); the label simply doesn't guarantee the property the class name implies. A column headed "size class" was showing 22 cm items as "oversize."

**The lesson:** when a class/tier name asserts a physical property (size, severity, risk) but is *derived from a categorical label* (a packagetype, a status string, a product code), the label is a proxy and will have members that violate the implied property. Cross-check against the physical measurement before trusting the class name — or the name misrepresents its members. The fix here was the hybrid: packagetype-family AND physically-large-or-unmeasured, which reclassified 98 parcels and removed the nonsensical tiny-oversize rows.

**How to apply:** I had actually flagged the "packagetype is a product-class flag, not a size guarantee; ~40% ship parcel" limitation earlier in the same session — and still shipped the pure-label class without the cross-check. Next time I name a class after a physical property, build the reality filter in from the start rather than waiting for the principal to surface a member that obviously violates it. Niklavs's "looks like DQ" was the right instinct pointed at the wrong cause (label semantics, not measurement) — confirm which kind before fixing.
