# S278 — 2026-06-19 — Cockpit terminal copy: selection-snapshot fix

- Niklavs reported he couldn't copy from the cockpit terminal — "only part of the text copies," and when copying text he'd typed into claude's own prompt, Ctrl+C wouldn't copy and **two Ctrl+C's erased the input**. Diagnosed evidence-first by reading the copy path rather than guessing: the backend `_write_clipboard_text` (`cockpit/backend.py`) handles arbitrary-length UTF-16 (no truncation), and `.term-host` has carried no CSS transform since [[S073_d82c4fbc_terminal-scroll-root-fix]] (clean 1:1 mouse→cell), so the write bridge and the old selection-drift were both ruled out before touching anything.
- **Root cause (pinned via the "Ctrl+C erased it" tell):** Ctrl+C only erases when it's read as SIGINT, which only happens when no selection is registered. Claude's live TUI repaints by clear-then-rewrite, and a repaint **clears xterm's live selection** — so by the time the user reaches Ctrl+C, `hasSelection()` is false, the copy branch is skipped, and `\x03` falls through to claude, wiping its input line. Same family explains "only the last part copies" (only repaint-surviving rows remain selected).
- **Fix (all in [[term.js|cockpit/web/term.js]], code-only):** added a selection snapshot — `_selCache`/`_selCacheAt` populated by a new `term.onSelectionChange` handler that captures the full selected text the instant it's made (ignoring the empty events a repaint emits). The Ctrl+C handler now copies the live selection if present, **else** the snapshot if recent (≤5s), so a Ctrl+C right after selecting always copies and never SIGINTs. Snapshot is consumed on copy and cleared on typing (`_handleData`) so a genuine interrupt still works: first Ctrl+C after a select = copy, the next one (or after typing) = interrupt.
- **Gates:** `node --check cockpit/web/term.js` clean; `cockpit/test_backend.py` 79/79 (clipboard endpoint untouched). Confirmed `onSelectionChange` + `clearSelection` exist in the vendored xterm before relying on them.
