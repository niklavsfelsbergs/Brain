- **Diagnosed the cockpit terminal selection/scroll drift.** Symptom: dragging a selection in the embedded xterm highlighted ~10 rows *below* the pointer (worse the lower you click) and you couldn't scroll to the true bottom. Root cause: the whole UI renders under CSS `zoom: 1.35` (`.app-grid`), and xterm.js maps mouse→cell in true pixels — a scaled ancestor makes the math drift multiplicatively and the FitAddon miscount rows. DOM panels tolerate zoom; only xterm breaks.
- **First fix broke the layout → reverted precisely.** A counter-`zoom` on the terminal host (`zoom: 0.7407` + `width: calc(100% * --zoom)`) blew the terminal out to 135% width with a giant font in WebView2 — the zoom-cancel didn't reduce the layout footprint there. Reverted my three edits by reversing each Edit (NOT `git checkout`: [[cockpit/web/term.js]]/[[cockpit/web/styles.css]] carried already-committed sibling reskin work, and `main.js` had unrelated uncommitted work). Verified `backend.py` serves `no-store`, ruling out stale CSS as the cause.
- **Second fix landed (principal confirmed "we're good").** Graceful-failure design: outer `.term-frame` is a plain flex item with `overflow:hidden` so the column layout is structurally protected; inner `.term-host` counter-scales via `transform: scale(1/--zoom)` — visual-only, so it can't disturb layout footprint the way `zoom` did — bringing the terminal to net scale 1.0. Font `13→18` to preserve the prior visual size. [[cockpit/web/term.js]] + [[cockpit/web/styles.css]].
